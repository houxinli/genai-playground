# 2026-04-07 翻译流水线状态固化与 P0 可靠性收敛

## 背景

这轮工作来自一次完整的 Pixiv 全流水线实战：先下载，再翻译，再合并打包。跑通之后，暴露出几个工程性短板：

- 半成品双语文件会留在目录里，后续任务容易把它们误认为已完成。
- 打包过度依赖译后 YAML，标题缺失或漂移时会直接反映到最终产物。
- 进展、组件状态和下一步计划主要散落在对话里，不利于新的 agent 或新的会话快速接手。

## 本轮落地

### 工程改动

- 新增翻译运行状态持久化，输出现在会被区分为 `missing/partial/running/failed/complete`。
- 翻译任务规划会识别半成品和失败产物，并自动计划重跑。
- 打包链路新增 `.meta.json` / `index.json` 元数据回退。
- 修正了 quality checker 中 `bilingual` 参数链路的运行时问题。

### 测试与验证

- 新增 `file_handler`、`extract_chinese`、`quality_checker` 的回归测试。
- 清理了与当前 QC prompt 合同不一致的旧测试。
- 当前基线为：

```bash
conda run -n llm python -m unittest discover -s tasks/translation/src -t . -p "*_test.py"
```

结果：25 个测试通过。

### 文档改造

- 新增 [`../PROJECT_STATUS.md`](../PROJECT_STATUS.md) 作为当前项目状态、组件状态和开发计划的真相源。
- 更新了根 `README.md`、`docs/README.md`、`docs/AGENT_CONTEXT.md`、`tasks/translation/README.md`、`AGENTS.md`，让新 agent 更容易找到当前上下文。

## 当前判断

翻译项目现在已经从“能跑”进入“可以继续系统化工程化”的阶段。P0 解决的是可靠性和可恢复性，接下来优先级最高的仍然是：

1. 内建并发调度。
2. repair 流程一体化。
3. token-aware batching。
4. 更强的自动质量 gate。

## 维护约定

- `PROJECT_STATUS.md` 维护当前真相。
- `AGENT_CONTEXT.md` 保留稳定背景和使用习惯。
- Journal 继续记录关键变更和决策，而不是承载当前任务清单。
