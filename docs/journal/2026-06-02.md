# 2026-06-02 修复破坏性 repair + 收尾 pixiv 53230930

## 背景

检查 pixiv 53230930 的流水线状态时发现：下载✓、翻译✓（`_bilingual/` 完整无标记），但**修复阶段把输出弄坏了**——`_bilingual_fixed/` 8 个文件各有 7–26 个 `[翻译未完成]`，且没有打包。

## 根因

`tasks/translation/src/core/partial_translator.py`：repair 重译某行失败/被拒/校验不过时，会写入 `[翻译未完成]` 并覆盖 `reference_translations[idx]`，把原有的好译文一起丢掉。该语料是 R-18，修复用的模型拒绝大量露骨/拟声词行，于是好译文被批量替换成占位符——repair 比不修还差。

## 落地

### 代码（分支 `fix/repair-non-destructive`）

- repair 改为**非破坏性**：重译不可用时沿用既有的任何非空原译，只有原本就为空才标占位符。任一行只会"改好或保持"，绝不引入更差的占位符。
- 新增 `partial_translator_test.py`（6 个用例）。套件 44 → 50 全绿。

### 数据收尾（grok-4.3）

- 发现 preset 钉死的 `x-ai/grok-4.1-fast` 已下架（OpenRouter 现有 grok-4.3 / 4.20）。改用 `x-ai/grok-4.3` 重修。
- 重修后：8 文件 **0 占位符**，QA error 总数 761 → 613，7/8 文件下降，无文件回退。
- 打包：`extract_chinese.py` 提取中文到 `53230930_zh/`（8 篇）。

## 发现的优化点（按性价比）

1. **repair 非破坏性**（已修）。
2. **确定性词表替换优先于 LLM**：最终 zh 包仍有 45% 行残留假名，集中在极少数复现外来词（おっぱい×70、ブラ×27、ザーメン×14…）。LLM repair 反复漏译这类词；用 `scripts/apply_translation_replacements.py` 一个词表即可批量清掉，便宜且无拒译风险。
3. **repair 模型须与翻译同档（不审查）**：R-18 语料别用更保守模型做 repair。本项目有 heretic MLX preset。
4. **QA gate 分级**：纯拟声词/音效行别一律判 hard error，减少无意义 repair churn。
5. **repair 护栏**：repair 后再跑 QA，error 数比修复前变多则拒绝写回。
6. **preset 模型 slug 会过期**：需要校验/更新机制（这次 grok-4.1-fast 404）。
7. **目录治理**：实验产物目录堆积（如 50235390 的 6 个变体）。
