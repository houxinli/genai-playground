# 2026-06-12 翻译系统目标架构收敛

## 背景

现有流水线已经能下载、翻译、QA、局部修复和打包，但双语 TXT 同时承担结果、checkpoint、repair 输入、
QA 输入和发布物。随着人名一致性、跨文本知识、多模型候选、Agent review 和用户句级反馈加入，这种模型会
持续放大路径推断、文本重新对齐、目录版本和并发状态的复杂度。

## 决策

- JSON 保存不可变、可移植、可交给 Codex/Claude Code/Cursor 的规范业务工件。
- SQLite/WAL 保存可重建索引、review queue、worker lease、运行状态与指标。
- API、本地模型、Agent harness 和人工编辑统一作为 candidate producer。
- 每个 segment 可以保留多个 candidate；`DocumentVersion` 保存选择结果。
- bilingual/zh/package 改为从确定版本渲染，不再作为状态真相源。
- 用户标记问题创建 annotation，再生成定向 review/repair task。
- repair 只生成候选，新结果通过 QA 和比较后才可能成为新版本。
- 人名与术语升级为有作用域、有 authority、有版本快照的 entity/knowledge 模型。

完整设计见
[`../../tasks/translation/docs/system-design.md`](../../tasks/translation/docs/system-design.md)。

## 迁移原则

1. 先冻结 schema、fixture 和 golden output。
2. 先引入 Document/Segment 与 renderer shadow path，不立即替换现有生产入口。
3. 导入现有 bilingual/fixed 为 legacy candidate，历史结果不丢失。
4. candidate/version 稳定后再接 QA/repair 和 Agent harness。
5. SQLite 与多 worker 最后加入，且必须支持从 JSON 重建。
6. 每个阶段保留明确回滚路径，禁止一次性重写 `pipeline.py`。

## 验证

文档更新前重新运行：

```bash
conda run -n llm python -m unittest discover -s tasks/translation/src -t . -p "*_test.py"
```

结果：50 个测试通过。

## 跨 Agent 开发连续性

### 背景

Codex、Claude Code、Cursor 各自拥有独立聊天上下文。仅靠“继续”无法可靠判断当前 branch、已完成步骤、
未提交改动和下一动作，也无法区分项目 roadmap、翻译业务 job 与开发任务执行游标。

### 决策

- 用 `agent/tasks/<task-id>/state.json` 保存 branch-level 开发任务执行游标。
- 用 `checkpoints.jsonl` 保存 append-only 交接记录。
- “继续”固定执行：定位任务、加载最小 context、校准 Git、执行 `next_action`、验证、更新 checkpoint。
- “交接”不开始新步骤，只固化当前状态、验证结果、未提交改动和下一动作。
- Codex、Claude Code、Cursor 共用同一协议；harness 配置只保留指针，不复制规则。
- GitHub Issue 保存目标和验收标准，PR 保存 diff/review/CI，仓库 task state 保存精细执行游标。
- 同一 branch/worktree 只允许顺序交替；并行工作必须拆 branch/worktree 和 task state。
- 旧的混合工作区不回填虚构 active state；从下一条干净 topic branch 开始启用。
- 增加 task/checkpoint validator，检查 schema、task-id/目录、branch 唯一性、plan 引用和 checkpoint 尾指针。
- validator 已接入 Make 与 GitHub Actions，并增加 7 个单元测试。

协议见 [`../AGENT_WORKFLOW.md`](../AGENT_WORKFLOW.md)，schema 与模板见 [`../../agent/`](../../agent/)。
